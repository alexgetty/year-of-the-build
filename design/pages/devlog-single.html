Week 08 February 25, 2025 Pocket Greenhouse

The Great Refactor

Sometimes you have to burn it down to build it right. This week I scrapped the component library and started fresh with a cleaner architecture. Here is what I learned about technical debt, when to refactor, and why the second system is often better.

8 min read 1,247 words

Week eight of Year of the Build, and I made a decision that felt equal parts terrifying and liberating: I deleted the entire component library I had built for Pocket Greenhouse and started over.

Not because it was broken. The components worked. Users could add plants, set reminders, and track growth. But every time I tried to add a new feature, I found myself fighting the architecture. The code was fighting back.

The Moment of Clarity

It happened on Tuesday. I was trying to add a simple toggle for dark mode support. What should have been a 30-minute task turned into a four-hour descent into CSS specificity hell. Every component had hardcoded colors. Every button had slightly different padding. The spacing scale was a lie I told myself.

Technical debt is not a loan. It is a bet against your future self. And your future self always has to pay up, with interest.

My internal monologue at 2 AM

I closed my laptop, went for a walk, and came back with a decision: this codebase needed to die so the project could live.

What Went Wrong

Looking back, the problems were predictable. I had rushed the initial architecture because I wanted to ship fast. Fair enough for a 12-project year. But I had made specific mistakes that compounded over time:

  1. No design tokens. Colors, spacing, and typography were scattered across files with no single source of truth.
  2. Component API inconsistency. Some buttons took a variant prop, others took type. Some cards had padding built in, others expected wrapper elements.
  3. Over-abstraction in the wrong places. I had a BaseInput component with twelve props when I only ever used three configurations.
  4. Under-abstraction where it mattered. Layout primitives did not exist. Every page reinvented spacing.
Warning

If you find yourself copy-pasting the same layout code more than twice, stop. Create a layout component. Future you will thank present you.

The Rebuild Strategy

I gave myself two days for the rebuild. Aggressive, but possible because I knew exactly what I was building this time. Here was the plan:

Day One: Foundation

Design tokens first. Every color, every spacing value, every font size lives in a single file. If it is not in the tokens, it does not exist.

TypeScript 
// tokens.ts - The single source of truth
export const tokens = {
  colors: {
    paper: '#FDFBF7',
    ink: '#1C1917',
    forest: '#2E5A3C',
    amber: '#B45309',
  },
  space: {
    1: '8px',
    2: '13px',
    3: '21px',
    4: '34px',
    5: '55px',
  },
  text: {
    xs: '10px',
    sm: '13px',
    base: '16px',
    lg: '26px',
    xl: '42px',
  }
} as const;

Day Two: Components

Only build what you need. I started with six primitives:

  • Stack - Vertical layout with consistent gaps
  • Row - Horizontal layout with alignment options
  • Text - Typography component with size variants
  • Button - Three variants, three sizes, done
  • Card - Container with optional padding
  • Input - Text input with label and error state

That is it. Six components. Every page in Pocket Greenhouse is built from these six primitives plus page-specific components.

Note

The constraint of only six base components forced me to think carefully about composition. Most "complex" UI is just primitives stacked together thoughtfully.

The Result

Two days later, I had a new component library. The total line count dropped by 40%. Adding dark mode took 15 minutes.

The new architecture is simpler, faster to develop with, and easier to maintain.

More importantly, I enjoy working on the codebase again. That matters. When the code feels good to write, you write more of it. When it feels like a chore, you avoid it.

When to Refactor

Not every messy codebase needs a complete rewrite. Here is my new rule of thumb:

  1. Can you still ship features? If yes, keep shipping. Refactor incrementally.
  2. Are new features taking 3x longer than they should? Consider a targeted refactor of the problem area.
  3. Do you dread opening the codebase? Something is wrong. Fix it before it kills the project.

For Pocket Greenhouse, all three were true. The refactor was overdue.

Next Week

With the new architecture in place, I am finally adding the features I have been putting off: dark mode, push notifications for watering reminders, and a plant health dashboard. The roadmap is clear, the code is clean, and I am excited to build again.

Sometimes you have to take a step back to take two steps forward. This was one of those weeks.
