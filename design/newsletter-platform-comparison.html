Newsletter Platform Research: YOTB

Automated newsletter via GitHub Action, MDX source, 52 sends/year, 100-1000 subscribers

The Short Version

Buttondown is the right call - but not for the reasons you might assume. The real competition is Beehiiv (surprisingly generous free tier) and Resend (if you want pure API minimalism). Self-hosting with Listmonk is viable but costs more in time than it saves in money at this scale.

Skip: Mailchimp (gutted free tier), ConvertKit (overbuilt), Loops (SaaS-focused pricing), raw AWS SES (too much plumbing).

Context: What We're Solving For

Weekly devlog, same MDX content powering site and email. Push to main triggers GitHub Action, which calls newsletter API. No manual intervention. Growing from zero to (optimistically) 1,000 subscribers over 12 months.

Hard requirements:

  • API-first (POST endpoint for sending)
  • Can accept rendered HTML or markdown
  • Reliable deliverability on shared infrastructure
  • Cost-effective at indie scale

Deep Dive: The Real Options

Managed Buttondown

The indie darling. One person (Justin Duke) runs it. API does everything the UI does. Plain-text-forward templates help deliverability. 62 pre-flight checks before each send.

  • API access on all plans (2025 change)
  • REST API matches full UI capability
  • Privacy-first, no tracking by default
  • Markdown native
  • 100 sub free tier is tight
  • No tier between 100 and 1,000

Sources: Pricing, API Docs, 2025 Update

Free Tier Beehiiv

The growth-focused newsletter platform. Surprisingly generous free tier: 2,500 subscribers, unlimited sends, AND API access. Built for people who want to monetize newsletters.

  • 2,500 subs free (covers your whole first year)
  • API access on free tier
  • Strong automation via Make/Zapier
  • Growth features you won't use (referrals, ads)
  • More newsletter platform than email API
  • April 2024 price increases

Sources: Pricing, Pricing FAQ

Developer Resend

The developer's email API. Built by the React Email team. Primary focus is transactional, but marketing/newsletter feature exists. Cleanest API of the bunch.

  • 1,000 contacts free, unlimited sends
  • Best-in-class developer experience
  • React Email integration
  • Marketing feature is newer/less mature
  • Built for transactional first
  • Less newsletter-specific features

Sources: Pricing, Pricing Guide

Self-Hosted Listmonk

Open source, single binary, handles millions of emails. Used by Zerodha in production. But: you own everything - infrastructure, updates, deliverability, backups.

  • Zero software cost
  • Unlimited everything
  • Full control, full data ownership
  • Need VPS ($5-10/mo) + SMTP provider
  • PostgreSQL, Docker, Nginx required
  • You manage deliverability reputation
  • No vendor support

Sources: Official Site, GitHub

The Self-Hosting Trap

Listmonk looks cheap on paper, but factor in: initial setup (4-8 hours), ongoing maintenance, deliverability monitoring, backup management, security patches. At 52 emails/year to under 1,000 people, you're spending more in time than you're saving in money. Self-hosting makes sense at 10K+ subscribers or when you need capabilities no managed service offers.

The Question Behind the Question

Why was Buttondown the assumed choice? Let me challenge it:

Arguments For Buttondown

  • Philosophy match: Indie project, solo maintainer, privacy-first, markdown-native. It's YOTB's vibe.
  • API quality: Everything the UI does, the API does. No second-class citizen treatment.
  • Deliverability: Plain-text-forward templates, 62 pre-flight checks, good inbox placement.
  • Predictable pricing: $9/mo for 1K is simple. No overage surprises.

Arguments Against Buttondown

  • Free tier is tight: 100 subscribers. You'll be paying within months.
  • Beehiiv is free longer: 2,500 subs free with API access. That's potentially your entire first year at $0.
  • Resend is more minimal: If you just want "POST to send email," Resend's API is cleaner.
  • Single maintainer risk: Justin Duke is one person. What happens if he burns out?

The Real Tradeoff

Beehiiv saves you $108 in year one. But Beehiiv is a newsletter platform - growth tools, referral programs, ad network. You're not using any of that. You're just hitting an API. The overhead is conceptual, not financial, but it's there.

Resend is pure API, but it's built for transactional email first. Newsletter/marketing is a newer feature. You'd be an early adopter of that capability.

Recommendation: Buttondown

Not because it's cheapest (it's not). Because it's the right tool for this job:

  • Markdown-native matches your MDX workflow
  • API-first philosophy means automation isn't an afterthought
  • Plain-text-forward matches devlog aesthetic and helps deliverability
  • Predictable costs - $9/mo is budgetable, no surprises
  • Values alignment - indie supporting indie

If cost is truly the constraint: Start with Beehiiv's free tier. Migrate to Buttondown when you hit 2,500 or when Beehiiv's platform-ness annoys you. Both have export capabilities.

If you want the simplest API: Resend. Accept that newsletter is their second priority.

If you want maximum control: Listmonk + AWS SES. Accept the time cost.

Implementation Notes: Buttondown + GitHub Action

The integration is straightforward. Buttondown's API accepts:

  • POST /v1/emails - Create and optionally send an email
  • Body can be markdown or HTML
  • Subject from frontmatter, body from MDX content

GitHub Action workflow:

  1. Detect new file in /content/devlogs/
  2. Parse frontmatter (check draft: false)
  3. Extract title for subject, render MDX to HTML for body
  4. POST to Buttondown API

Rate limits: 100 requests/hour on subscriber endpoints, no documented limit on email sending for normal usage. 52 weekly sends is trivial.

What I'd Watch

  • Beehiiv's API maturity: They're focused on being a platform, not an API. Check if their API supports your exact workflow before dismissing.
  • Resend's marketing feature: If it matures, it could be the cleanest option. Worth checking in 6 months.
  • Buttondown's single-maintainer model: Not a dealbreaker, but worth knowing. The codebase isn't open source.
